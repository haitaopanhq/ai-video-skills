# 选题拆解与图文连载生成 Skill

## 名称

选题拆解与图文连载生成 Skill

## 适用场景

当用户给定一个具体选题，希望围绕该选题完成：

- 系列主题拆分
- 图文标题规划
- 长文标题规划
- 每日排期
- 周计划
- 周总结
- 图文关键词提炼
- 图文内容概要
- 文章结构大纲
- 批量生成图文提示词
- 按模板风格生成图片
- 每批不超过 10 张图文

本 Skill 用于把一个大主题拆成一组有节奏、有主线、有流量入口的内容连载。

## 核心原则

### 1. 先建立总叙事线

不要直接堆标题。先把选题压成一条清晰主线。

格式：

```text
总主题：
《XXX》第一季 / 系列名

核心叙事线：
A → B → C → D → E → F → G

核心判断：
这个系列不是讲一堆孤立知识点，而是在讲一条演进逻辑。
```

示例：

```text
《云上帝国》第一季
从闲置服务器到 Linux 统治云时代

主线：
资源困局 → 虚拟化革命 → 路线战争 → 云平台 → 存储暗线 → 网络暗线 → Linux 统治云时代
```

### 2. 每天只保留一个主矛盾

每天不要塞太多概念。

错误方式：

```text
CPU、预算、SLA、超卖、调度、KVM、AWS 全部一天讲
```

正确方式：

```text
一天一个主矛盾：
资源为什么卡？
物理服务器能不能被切开？
谁掌控资源入口？
云平台真正卖什么？
数据怎么办？
网络如何跟上？
Linux 为什么成为底座？
```

### 3. 图文负责拉流量，长文负责沉淀认知

每天结构建议：

- 图文 1：现象钩子
- 图文 2：机制拆解
- 图文 3：收藏型知识卡
- 长文：当天主矛盾完整展开

图文标题适合：

- 一图详解：XXX
- 一图看懂：XXX
- 为什么 XXX？
- XXX 背后真正发生了什么？

长文标题适合：

- XXX 为什么不是改良，而是一场革命
- XXX 为什么开始接管基础设施入口
- XXX 为什么最终赢下资源控制权战争
- 3 分钟速览《XXX》：从 A 到 B

### 4. 一周连载要像连续剧，不要像知识清单

优先使用这种节奏：

- 周一：提出总矛盾
- 周二：技术转折
- 周三：路线冲突
- 周四：平台化
- 周五：隐藏暗线一
- 周六：隐藏暗线二
- 周日：总收束

也可以根据内容密度压缩为：

- 周一：开篇
- 周三：推进
- 周五：高潮
- 周日：总篇

不要为了日更而凑数。如果几个主题本质上是一条因果链，应该压缩成一天。

## 标准工作流

### Step 1：接收用户选题

输入示例：

```text
选题：云上帝国第一季，从闲置服务器到 Linux 统治云时代
```

需要提取：

- 主题名
- 目标读者
- 内容平台
- 周期长度
- 是否需要图文
- 是否需要长文
- 是否已有图片模板
- 是否要求每日 2-3 篇图文
- 是否需要批量生成图片

如果用户没有说明，默认：

- 周期：7 天
- 每天：2-3 个图文 + 1 篇主文
- 风格：技术文明史 / 漫谈式 / 工程师友好
- 图文：蓝白科技风信息图
- 每次生成图片：不超过 10 张

### Step 2：拆分总主线

输出格式：

```markdown
# 系列总主题

## 总标题
《XXX》第一季：XXX

## 核心钩子
一句话说明这个系列解决什么问题。

## 总叙事线
A → B → C → D → E → F → G

## 明线
XXX

## 暗线
XXX
```

示例：

```markdown
## 总叙事线

资源困局 → 虚拟化革命 → 路线战争 → 云平台 → 存储暗线 → 网络暗线 → Linux 统治云时代

## 明线

计算资源如何从闲置、冲突、调度，走向虚拟化与云平台。

## 暗线

存储如何从设备走向 API。
网络如何从物理连接走向软件定义。
```

### Step 3：生成周排期表

输出格式：

```markdown
| 日期 | 主题 | 图文 1 | 图文 2 | 图文 3 | 长文 |
|---|---|---|---|---|---|
| 周一 | 主题 A | 标题 | 标题 | 标题 | 长文标题 |
| 周二 | 主题 B | 标题 | 标题 | 标题 | 长文标题 |
```

标题统一带系列感。

图文标题建议统一为：

```text
一图详解：XXX
```

长文标题不必都带“一图详解”，要更像文章。

### Step 4：补全每篇图文规划

每篇图文输出：

```markdown
## 图文标题

### 关键词
关键词 1、关键词 2、关键词 3、关键词 4、关键词 5

### 封面金句
一句强判断。

### 图文内容概要
1. 第一层
2. 第二层
3. 第三层
4. 第四层
5. 核心结论

### 配图方向
蓝白科技风 / 信息图 / 5 层结构 / 图标 / 箭头 / 总结区
```

示例：

```markdown
## 一图详解：资源过剩时代的窘境

### 关键词
服务器闲置、资源错配、低利用率、机房成本、预算浪费

### 封面金句
企业不是没有资源，而是资源总是不在需要的地方。

### 图文内容概要
1. 服务器越来越强
2. 企业采购越来越多
3. 大量资源长期闲置
4. 热点业务仍然缺资源
5. 本质是资源错配与调度失灵
```

### Step 5：补全长文概要

每篇长文输出：

```markdown
## 长文标题

### 长文关键词
XXX、XXX、XXX

### 核心观点
一句话讲清楚这篇文章的判断。

### 文章结构
#### 01 开场
#### 02 背景
#### 03 核心矛盾
#### 04 技术转折
#### 05 影响
#### 06 收束
#### 07 引到下一篇

### 结尾钩子
一句话引到下一篇。
```

长文风格：

- 娓娓道来
- 技术文明史感
- 工程师读得懂
- 不要术语堆砌
- 用时代转场串联

## 标题生成规则

### 图文标题规则

优先使用：

- 一图详解：XXX
- IT简史｜图解：XXX
- 一图看懂：XXX
- 为什么 XXX
- XXX 到底在争什么
- XXX 如何重新定义 XXX

如果是技术史、基础设施演进、云原生、AI Infra、Linux、网络、数据库、安全等系列，优先使用稳定栏目结构：

- 图文首篇：`一图详解：XXX`
- 图文补充篇：`IT简史｜图解：XXX`
- 长文篇：`IT简史｜漫谈：XXX`

栏目分工：

- `一图详解` 负责当天最核心、最适合做封面的主题，用来吸引点击。
- `IT简史｜图解` 负责机制拆解、概念说明、路线对比、知识卡片。
- `IT简史｜漫谈` 负责公众号主文，承接历史叙事和技术判断。

每天只保留第一篇图文使用 `一图详解`，其余图文统一使用 `IT简史｜图解`，长文统一使用 `IT简史｜漫谈`，形成稳定栏目感。

适合流量的标题结构：

- 一图详解：为什么企业服务器长期只有 10% 利用率
- 一图详解：Hypervisor 不是为了虚拟，而是为了分资源
- 一图详解：KVM 为什么最终赢了 Xen
- 一图详解：S3 为什么让存储变成 API
- 一图详解：eBPF 为什么把 Linux 推向控制面中心
- IT简史｜图解：Hypervisor 分资源
- IT简史｜图解：对象存储崛起
- IT简史｜图解：eBPF 走向控制面

### 长文标题规则

长文标题更适合：

- IT简史｜漫谈：XXX
- XXX 为什么不是改良，而是一场革命
- XXX 为什么开始吞掉基础设施入口
- XXX 为什么最终赢下资源控制权战争
- 3 分钟速览《XXX》：从 A 到 B

示例：

- 资源不缺，为什么系统还是会卡
- IT简史｜漫谈：系统为何会卡
- 虚拟化为什么不是改良，而是一场资源革命
- IT简史｜漫谈：虚拟化资源革命
- KVM 为什么最终赢下资源控制权战争
- IT简史｜漫谈：KVM 赢得控制权
- 云平台为什么开始吞掉基础设施入口
- IT简史｜漫谈：云吞掉基础设施
- S3 为什么让存储变成了云时代的地基

## 内容风格要求

### 文章风格

- 像“漫谈技术文明史”
- 不要写成文档说明书
- 不要堆术语
- 每一节都要有时代转场
- 把技术写成演进故事
- 技术细节够准确，但表达要能让工程师和泛技术读者读下去

### 常用表达

- 很多技术，刚出现时并不宏大。
- 它们只是为了解决一个朴素的问题。
- 表面看，是 XXX。
- 更深一层看，是 XXX。
- 真正改变世界的，不是 XXX，而是 XXX。
- 这不是一组孤立技术点。
- 这是一条演进线索。
- 从这一刻开始，XXX 不再只是 XXX，而开始成为 XXX。

## 执行约束

- 不要一次生成超过 10 张图片。
- 如果用户上传参考图，必须沿用参考图风格。
- 图文标题必须服务主线，不要只追热点。
- 每天只能有一个主矛盾。
- 技术史类系列默认使用栏目化标题结构：首篇 `一图详解`，补充图文 `IT简史｜图解`，长文 `IT简史｜漫谈`。
- 图文可以多角度，但不能把当天主题打散。
- 长文要负责承接上下文。
- 周日总篇要把整周内容收束成一条完整故事线。
- 隐藏暗线可以穿插，但不要抢主线。
- 如果内容过密，优先压缩，不要凑数。
- 每个图文都要有：标题、关键词、金句、5 层概要、核心结论。
